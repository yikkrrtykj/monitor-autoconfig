# TODO（功能与现场验收清单）

更新时间：2026-07-21

这份清单根据当前代码和此前讨论重新整理。原则是优先保证现场功能可靠，不为了“功能多”持续轮询交换机；涉及带宽、Telnet 或 SNMP 的操作默认按需执行。

## P0：合并或上服务器前必须确认

- [ ] **在真实 Cisco 核心交换机上验收 DHCP 面板**
  - 至少覆盖常用的 C9300 和 C3850；如果现场仍可能让 C2960 承担 DHCP，再补测 C2960。
  - 保存 Telnet 配置并执行连接测试，确认普通密码、用户名 + 密码、Enable 密码三种现场登录方式。
  - 对照交换机原始输出核对地址池总量、排除地址、已租用、剩余、使用率和冲突地址。
  - 记录一次采集耗时以及采集前后的交换机 CPU；确认 60 秒可见页刷新和 30 秒后台保护间隔不会造成明显波动。
  - 继续禁止定时执行 `show ip dhcp binding`，避免读取完整租约表。

- [ ] **在真实防火墙上验收 ISP 监控（本轮合并新增，务必现场确认）**
  - 山石 HA + WatchGuard HA 各验收一次；确认 `firewall-snmp` 采集到 WAN 口流量。
  - 确认 ISP 网关自动发现产出正确目标：看 topology-collector 的 `[isp-discovery]` 日志，网关与运营商名对得上；双线场景确认 `电信-1/电信-2` 命名与各自带宽阈值一一对应。
  - 故意拔一台物理防火墙（不是 VIP），确认 10 秒级收到单机离线告警（`infra-fw-unit-ping`），插回收到恢复；VIP 全程不误报。
  - 换防火墙 / HA 主备切换 / 重启时，确认只出现 `ignore counter glitch` 日志而**没有**几十 Gbps 的假带宽告警卡。
  - 关闭防火墙 SNMP 或改到不匹配 `FIREWALL_WAN_IF_FILTER` 的接口名，持续 2 分钟，确认收到“外网流量采集中断”告警，恢复后收到恢复卡。
  - 确认带宽绑定正确：按 WAN 口名/运营商名核对 200/500 等阈值落在正确的口上。

- [ ] **在一台干净的 Linux 服务器上做完整部署验收**
  - 从仓库重新构建，而不是复用本机旧镜像。
  - 确认 `platform-api` 镜像内同时包含 Python 3.13、`telnetlib3` 和 `iperf3`，宿主机不需要单独安装 iPerf3。
  - 确认 Docker 重启后赛事配置、DHCP 登录信息、告警状态和历史数据仍在。
  - 再做一次离线包安装，确认无外网现场只需预装 Docker、Compose 插件和 Python 3。

- [ ] **做一次完整的现场告警回归**
  - 新设备上线、设备离线、普通恢复、端口/链路、广播或组播风暴、LibreNMS 告警桥接各测一次。
  - 验证设备离线满 48 小时后旧周期被清理；再次出现只发送“新设备上线”，不发送旧离线恢复。
  - 验证清理停用设备时不会误删仍在使用的 LibreNMS 设备。
  - 正式环境不要继续使用之前公开在聊天中的测试 Webhook；合并前换成正式机器人地址。

- [ ] **从实际项目出口手动验证公共 iPerf3 节点**
  - 分别检查香港、新加坡、土耳其和印度尼西亚至少一个节点能否连接。
  - 公共节点没有可用性保证，验收时记录可用服务器、端口和日期；不能因为本机成功就认为所有项目现场都成功。
  - 只在空闲窗口测速，避免占满赛事公网出口。

## P1：下一轮优先改进

- [ ] **为 iPerf3 增加“停止测速”**
  - 当前页面的“取消”只取消开始前的确认，任务开始后只能等待完成或最长 60 秒超时。
  - 后端改为可管理的子进程，并提供只终止当前测速任务的接口；页面显示明确的“停止测速”按钮。

- [ ] **为 iPerf3 任务增加任务编号，隔离多个页面的状态**
  - 当前平台同一时间只允许一项测速，这是合理的，但进度状态是全局的。
  - 为每次任务生成 ID，避免两个浏览器标签、刷新后的旧请求或另一位操作员串用同一份进度和结果。

- [ ] **保存少量 iPerf3 历史并支持赛前比较**
  - 建议只保留最近 10～20 次：时间、地区、服务器、端口、方向、平均速率、总传输量、TCP 重传和每秒数据。
  - 用于比较不同项目/不同出口的赛前体验；不做长期高频测速，也不自动定时占用带宽。

- [ ] **把公共 iPerf3 节点列表从大段前端代码中独立出来**
  - 节点地址和端口会失效，改成单独的版本化 JSON/配置文件更方便维护。
  - 页面显示“最后核验日期”，并提供轻量的 DNS/TCP 可达性检查；可达性检查不等同于带宽测试。
  - 同一地区首选节点失败时，可提示切换到该地区的下一个节点，但不要未经确认自动连续打满多个服务器。

- [ ] **补齐 DHCP 的真实 IOS/IOS-XE 输出测试样本**
  - 把现场 C9300、C3850（以及需要时的 C2960）只读命令输出脱敏后保存为测试夹具。
  - 覆盖不同软件版本的提示符、分页、无冲突、空地址池、不支持统计命令和非 15 权限等情况。
  - 页面增加“上次成功采集时间、采集耗时、缓存/实时”提示，读取失败时保留上次成功数据并明确标为旧数据。

- [ ] **让 CI 同时验证生产使用的 Python 3.13**
  - 当前 GitHub CI 只跑 Python 3.11，实际 `platform-api` 镜像使用 Python 3.13。
  - 增加 Python 3.13 测试，确保真正走到 `telnetlib3` 分支；3.11 可继续保留作兼容性检查。

- [ ] **在真实 Linux 上增加一套轻量冒烟测试**
  - 检查 Compose 配置、容器健康、控制台登录、DHCP 设置保存、iPerf3 可执行文件以及飞书桥接健康状态。
  - 测试默认不连接交换机、不发正式飞书消息、不运行带宽测试；需要时再人工触发现场测试。

## P2：已讨论、后续再做

- [ ] **飞书手动查询光功率（不做自动光功率告警）**
  - 目标是按命令查询 LibreNMS 已采集的 Cisco 光模块 Tx/Rx dBm，并标出明显异常值。
  - 交换机到交换机按单模理解；交换机到防火墙的模块按多模理解；铜缆接口忽略。
  - 现有 Webhook 机器人只能被动发消息，若要在群里接收“查光功率”命令，需要飞书应用机器人、事件订阅和回调服务。
  - 默认只手动检查，不持续推送光功率告警，避免增加噪音。

- [ ] **应用/树莓派机器人**
  - 后续再考虑树莓派发布端在线、同步中、播放中、分配流和应用异常等状态。
  - 先明确应用提供的数据接口，再接入现有飞书告警；不要只靠频繁 Ping 或交换机 FDB 猜测业务状态。

- [ ] **组播业务的按需检查**
  - 大部分组播流量不经过核心，不能只读核心交换机就判断整条业务正常。
  - 后续先设计按需检查：IGMP Snooping、Querier、组播组和关键接入口；默认不做高频全网轮询。
  - 接入前先测 C2960 的 CPU 影响，超过可接受范围就只保留人工命令或延长采集周期。

- [ ] **拓扑和接口详情增强前先做负荷评估**
  - LibreNMS 继续承担交换机、端口、链路和基础性能监控。
  - 不额外高频读取全量 MAC/FDB、ARP 或接口明细；确需展示时采用页面打开后按需读取并缓存。

## 暂不做 / 保持现状

- 不迁移 DHCP 服务到 Kea/Stork；Cisco 核心继续提供 DHCP，当前页面只做只读展示。
- 不恢复“座位核对”页面。
- 不恢复“访问模式 / 外网访问 / 公网基础 URL”字段。
- 不自动持续运行 iPerf3，也不把公共测速结果当成运营商带宽验收的唯一依据。
- 暂不增加 UDP 打流；现阶段保持 TCP 上传/下载。需要排查抖动、丢包或组播时再单独设计 UDP 测试。
- 暂不强制把 Telnet 改成 SSH；内网现场可继续使用 Telnet，但账号权限尽量只满足只读命令。
- 不做自动光功率告警，先保留后续的手动查询方案。
- 不新增高频交换机轮询，尤其避免在 CPU 已较高的旧 C2960 上增加全量采集。
